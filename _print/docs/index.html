This section is where the user documentation for your project lives - all the information your users need to understand and successfully use your project.

For large documentation sets we recommend adding content under the headings in this section, though if some or all of them don’t apply to your project feel free to remove them or add your own. You can see an example of a smaller Docsy documentation site in the Docsy User Guide, which lives in the Docsy theme repo if you’d like to copy its docs section.

Other content such as marketing material, case studies, and community updates should live in the About and Community pages.

Find out how to use the Docsy theme in the Docsy User Guide. You can learn more about how to organize your documentation (and how we organized this site) in Organizing Your Content.

The Overview is where your users find out about your project. Depending on the size of your docset, you can have a separate overview page (like this one) or put your overview contents in the Documentation landing page (like in the Docsy User Guide).

Try answering these questions for your user in this page:

What is it?

Introduce your project, including what it does or lets you do, why you would use it, and its primary goal (and how it achieves it). This should be similar to your README description, though you can go into a little more detail here if you want.

Why do I want it?

Help your user know if your project will help them. Useful information can include:

  • What is it good for?: What types of problems does your project solve? What are the benefits of using it?

  • What is it not good for?: For example, point out situations that might intuitively seem suited for your project, but aren’t for some reason. Also mention known limitations, scaling issues, or anything else that might let your users know if the project is not for them.

  • What is it not yet good for?: Highlight any useful features that are coming soon.

Where should I go next?

Give your users next steps from the Overview. For example:

  • Getting Started: Get started with $project
  • Examples: Check out some example code!

4 - Concepts

What does your user need to understand about your project in order to use it - or potentially contribute to it?

This is a placeholder page that shows you how to use this template site.

For many projects, users may not need much information beyond the information in the Overview, so this section is optional. However if there are areas where your users will need a more detailed understanding of a given term or feature in order to do anything useful with your project (or to not make mistakes when using it) put that information in this section. For example, you may want to add some conceptual pages if you have a large project with many components and a complex architecture.

Remember to focus on what the user needs to know, not just what you think is interesting about your project! If they don’t need to understand your original design decisions to use or contribute to the project, don’t put them in, or include your design docs in your repo and link to them. Similarly, most users will probably need to know more about how features work when in use rather than how they are implemented. Consider a separate architecture page for more detailed implementation and system design information that potential project contributors can consult.

5 - Core Tasks

What can your user do with your project?

This is a placeholder page that shows you how to use this template site.

Think about your project’s features and use cases. Use these to choose your core tasks. Each granular use case (enable x, configure y) should have a corresponding tasks page or tasks page section. Users should be able to quickly refer to your core tasks when they need to find out how to do one specific thing, rather than having to look for the instructions in a bigger tutorial or example. Think of your tasks pages as a cookbook with different procedures your users can combine to create something more substantial.

You can give each task a page, or you can group related tasks together in a page, such as tasks related to a particular feature. As well as grouping related tasks in single pages, you can also group task pages in nested folders with an index page as an overview, as seen in this example site. Or if you have a small docset like the Docsy User Guide with no Tutorials or Concepts pages, consider adding your feature-specific pages at the top level of your docs rather than in a Tasks section.

Each task should give the user

  • The prerequisites for this task, if any (this can be specified at the top of a multi-task page if they’re the same for all the page’s tasks. “All these tasks assume that you understand….and that you have already….”).
  • What this task accomplishes.
  • Instructions for the task. If it involves editing a file, running a command, or writing code, provide code-formatted example snippets to show the user what to do! If there are multiple steps, provide them as a numbered list.
  • If appropriate, links to related concept, tutorial, or example pages.

7.1 - vfio vs igb_uio

vfio vs igb_uio

将驱动从 igb_uio 改为 vfio-pci 是DPDK发展中的一个重要趋势，主要区别和IOMMU的作用如下：

核心区别：igb_uio vs vfio-pci

特性igb_uio (Userspace I/O)vfio-pci (Virtual Function I/O)
内核支持依赖 UIO框架 和特定的 igb_uio.ko 内核模块。依赖现代、标准的 VFIO框架，内核原生支持（>= 3.6，完整功能需>=4.0）。
安全性/隔离性低。UIO直接将设备的全部内存映射暴露给用户空间，用户空间驱动（如DPDK）拥有对硬件的完全、无隔离的访问权限。一个错误的DPDK应用可能破坏系统。高。VFIO的核心功能是利用 IOMMU 为设备提供 DMA 地址翻译和访问保护。设备只能访问明确分配给它的内存区域，实现了用户空间与设备之间的安全隔离。
虚拟化支持不支持或不完善。很难安全地将一个物理设备直接分配给虚拟机（PCIe Passthrough）。原生支持。VFIO正是为安全、高效的硬件直接分配（如KVM的PCIe Passthrough）而设计的。这是它最大的优势之一。
依赖要求只需加载igb_uio模块，对系统配置要求低。必须启用IOMMU（在BIOS中开启VT-d/AMD-Vi，并在内核启动参数中启用intel_iommu=on或amd_iommu=on）。
发展状态DPDK早期采用的方案，逐渐被淘汰。当前推荐的标准方案，更安全、更现代，是DPDK和虚拟化生态的未来。

为什么必须在BIOS中开启IOMMU支持？

这是使用 vfio-pci 驱动 最根本的前提条件。原因如下：

  1. DMA保护与内存隔离 (核心原因)

    • 没有IOMMU时：设备进行DMA操作时，使用的是物理地址。一个拥有DMA能力的设备（如网卡）理论上可以读写系统内存的任何位置。如果用户空间的DPDK应用错误地配置了DMA地址，或者设备被恶意软件控制，可能导致系统崩溃或数据泄露。
    • 开启IOMMU后：IOMMU为每个设备（或设备组）维护一个 I/O 页表。当设备发起DMA请求（使用“I/O虚拟地址”（IOVA））时，IOMMU硬件会实时地将这个地址翻译成真正的物理地址。
    • 关键点：这个页表由内核的VFIO驱动严格管控。VFIO只会将DPDK应用申请并映射的特定内存区域的IOVA填入设备的I/O页表。因此，设备只能访问被明确允许的那部分内存，无法触及其他系统内存或内核内存，实现了完美的隔离。

  2. VFIO框架的基石

    • VFIO框架的设计哲学就是基于IOMMU的安全设备直接访问。它的整个权限控制、内存映射、设备隔离机制都建立在IOMMU硬件功能之上。没有IOMMU，VFIO就无法提供其核心的安全保障，也就失去了它相对于传统UIO的最大优势。

  3. 虚拟化直通 (PCIe Passthrough) 的必备条件

    • 将一块物理网卡直接分配给一个虚拟机（VM），让虚拟机独占该设备，这需要最严格的隔离。IOMMU确保：
      • 设备的DMA操作只能访问该虚拟机所属的内存，而不能访问宿主机或其他虚拟机的内存。
      • 设备产生的中断能被正确地重定向到对应的虚拟机。
    • 没有IOMMU，设备直通在安全上是不可行的。

简单比喻：

  • igb_uio：就像给了DPDK应用一把能打开整个房子所有房间的万能钥匙。它虽然方便，但一旦应用“发疯”，可以在房子里做任何事，非常危险。
  • vfio-pci + IOMMU：就像给DPDK应用分配了一个独立的公寓（IOMMU域），并只给它这个公寓的钥匙。应用只能在这个公寓内活动，无法进入其他房间（系统内存），也看不到其他公寓（其他虚拟机或进程的内存）。系统管理员（内核/VFIO）负责分配和管理这些公寓。

切换步骤与注意事项

  1. 检查并开启IOMMU:

    • 进入服务器BIOS，找到 Intel VT-d 或 AMD-Vi 选项，设置为 Enabled。
    • 修改Linux内核启动参数（通常在/etc/default/grub的GRUB_CMDLINE_LINUX中添加），加入：
      • Intel: intel_iommu=on iommu=pt
      • AMD: amd_iommu=on iommu=pt
      • iommu=pt 表示只为用于直通的设备启用IOMMU，对性能有轻微优化。
    • 更新grub并重启系统。

  2. 验证IOMMU是否生效:

    • 重启后执行 dmesg | grep -i iommu，应看到成功启用的日志。
    • 执行 cat /proc/cmdline 确认内核参数已生效。
    • 检查 /sys/class/iommu 目录是否存在。

  3. 绑定网卡到 vfio-pci:

    • 确保 vfio-pci 模块已加载 (modprobe vfio-pci)。
    • 使用DPDK的 dpdk-devbind.py 或 dpdk-devbind 工具进行绑定：
      dpdk-devbind.py -b vfio-pci 0000:01:00.0 0000:01:00.1
    • 或者使用 dpdk-setup.sh 脚本中的菜单选项来操作。

总结

需求场景推荐驱动理由
纯物理机部署，追求极简，且环境完全可控可以考虑 igb_uio配置简单，无需开启IOMMU。
生产环境、虚拟化环境、注重系统安全与稳定性、未来可能使用SR-IOV必须使用 vfio-pci提供硬件级别的DMA安全和内存隔离，是支持设备直通和现代虚拟化的基础。

简而言之，从 igb_uio 切换到 vfio-pci，是为了获得由 IOMMU 硬件提供的、至关重要的安全性提升和更好的虚拟化兼容性。而开启 BIOS 中的 IOMMU 支持，是启用这套安全机制的硬件开关。

这个问题很常见。IOMMU已开启但vfio-pci无法正常工作，而igb_uio可以，通常有以下几个原因。让我们一步步排查：

1. 检查当前网卡状态

# 查看网卡当前的驱动绑定
dpdk-devbind.py -s

# 或使用lspci查看
lspci -nnv -s 01:00.0 | grep "Kernel driver in use"

# 查看所有网卡的详细信息
lspci -Dvmmnn | grep -E "(Class|Device|Driver)"

2. 关键排查步骤

步骤1：确认IOMMU确实已启用

# 查看内核日志中的IOMMU信息
dmesg | grep -E "DMAR|IOMMU|AMD-Vi"

# 检查是否有IOMMU组
ls -ld /sys/kernel/iommu_groups/* 2>/dev/null | head -5

# 查看具体PCI设备的IOMMU组
ls -l /sys/bus/pci/devices/0000\:01\:00.0/iommu_group

# 查看DMAR表（Intel）
sudo cat /sys/firmware/acpi/tables/DMAR 2>/dev/null | hexdump -C | head -20

步骤2：检查vfio-pci模块是否正确加载

# 查看vfio相关模块
lsmod | grep vfio

# 如果没有加载，手动加载
sudo modprobe vfio
sudo modprobe vfio-pci
sudo modprobe vfio_iommu_type1

# 查看模块参数
cat /sys/module/vfio/parameters/enable_unsafe_noiommu_mode
# 应该显示 N 或 0

步骤3：绑定网卡到vfio-pci驱动

# 1. 先解除当前驱动绑定
sudo dpdk-devbind.py -u 0000:01:00.0 0000:01:00.1

# 2. 绑定到vfio-pci
sudo dpdk-devbind.py -b vfio-pci 0000:01:00.0 0000:01:00.1

# 3. 或者手动绑定（如果dpdk-devbind.py不工作）
echo "vfio-pci" | sudo tee /sys/bus/pci/devices/0000:01:00.0/driver_override
echo 0000:01:00.0 | sudo tee /sys/bus/pci/drivers/vfio-pci/bind

步骤4：检查vfio设备权限

# 查看/dev/vfio目录
ls -la /dev/vfio/

# 检查vfio组是否存在
getent group | grep vfio

# 将当前用户添加到vfio组
sudo usermod -aG vfio $USER

# 重新登录或使用newgrp
newgrp vfio

# 检查设备文件权限
ls -l /dev/vfio/*

# 应该看到类似：
# crw------- 1 root root  244,   0 Jan  1 00:00 /dev/vfio/vfio
# crw-rw-rw- 1 root vfio 244,   1 Jan  1 00:00 /dev/vfio/1

3. 常见问题及解决方案

问题1：IOMMU组包含多个设备

# 查看PCI设备的IOMMU组
ls -l /sys/bus/pci/devices/0000:01:00.0/iommu_group/devices/

# 查看整个组
ls -l /sys/kernel/iommu_groups/1/devices/  # 1是组号

解决方案：

  • 如果组内有多个设备，需要将所有设备都绑定到vfio-pci
  • 或者启用 acs_override（有安全风险）：
# 在内核参数中添加
GRUB_CMDLINE_LINUX="... intel_iommu=on iommu=pt pcie_acs_override=downstream,multifunction"

# 更新grub并重启
sudo update-grub
sudo reboot

问题2：Secure Boot阻止vfio-pci

# 检查Secure Boot状态
mokutil --sb-state

# 如果启用，可能需要禁用或签名模块
sudo mokutil --disable-validation

问题3：内核模块黑名单冲突

# 检查是否有其他驱动在抢设备
cat /etc/modprobe.d/* | grep -E "blacklist|(igb|ixgbe|i40e)"

# 将原生网卡驱动加入黑名单
echo "blacklist igb" | sudo tee -a /etc/modprobe.d/blacklist.conf
echo "blacklist ixgbe" | sudo tee -a /etc/modprobe.d/blacklist.conf
echo "blacklist i40e" | sudo tee -a /etc/modprobe.d/blacklist.conf

# 更新initramfs
sudo update-initramfs -u

问题4：DPDK编译配置问题

# 检查DPDK编译配置
cat /path/to/dpdk/build/.config | grep -E "VFIO|IGB_UIO"

# 重新编译DPDK，确保启用VFIO
cd /path/to/dpdk
meson configure build -Denable_driver_sdk=true -Denable_kmods=true
ninja -C build

4. 完整的排查脚本

#!/bin/bash
# check_vfio_issues.sh

echo "========== 1. 检查IOMMU状态 =========="
dmesg | grep -E "DMAR|IOMMU|AMD-Vi" | tail -10
echo ""

echo "========== 2. 检查内核参数 =========="
cat /proc/cmdline
echo ""

echo "========== 3. 检查vfio模块 =========="
lsmod | grep vfio
echo ""

echo "========== 4. 检查PCI设备 =========="
lspci -nn | grep -E "(Ethernet|Network)"
echo ""

echo "========== 5. 检查驱动绑定 =========="
dpdk-devbind.py -s
echo ""

echo "========== 6. 检查IOMMU组 =========="
for dev in $(lspci -D | grep Eth | awk '{print $1}'); do
    echo "设备 $dev:"
    if [ -d "/sys/bus/pci/devices/$dev/iommu_group" ]; then
        ls /sys/bus/pci/devices/$dev/iommu_group/devices/ | xargs -I {} lspci -s {}
    else
        echo "  未找到IOMMU组"
    fi
    echo ""
done

echo "========== 7. 检查vfio设备文件 =========="
ls -la /dev/vfio/
echo ""

echo "========== 8. 测试绑定 =========="
read -p "输入要测试的PCI设备地址（如01:00.0）: " pci_dev
full_dev="0000:$pci_dev"

echo "解绑设备..."
echo $full_dev | sudo tee /sys/bus/pci/devices/$full_dev/driver/unbind 2>/dev/null

echo "绑定到vfio-pci..."
echo "vfio-pci" | sudo tee /sys/bus/pci/devices/$full_dev/driver_override
echo $full_dev | sudo tee /sys/bus/pci/drivers/vfio-pci/bind 2>/dev/null

echo "检查结果..."
lspci -s $pci_dev -v | grep "Kernel driver"

8 - Contribution Guidelines

How to contribute to the docs

These basic sample guidelines assume that your Docsy site is deployed using Netlify and your files are stored in GitHub. You can use the guidelines “as is” or adapt them with your own instructions: for example, other deployment options, information about your doc project’s file structure, project-specific review guidelines, versioning guidelines, or any other information your users might find useful when updating your site. Kubeflow has a great example.

Don’t forget to link to your own doc repo rather than our example site! Also make sure users can find these guidelines from your doc repo README: either add them there and link to them from this page, add them here and link to them from the README, or include them in both locations.

We use Hugo to format and generate our website, the Docsy theme for styling and site structure, and Netlify to manage the deployment of the site. Hugo is an open-source static site generator that provides us with templates, content organisation in a standard directory structure, and a website generation engine. You write the pages in Markdown (or HTML if you want), and Hugo wraps them up into a website.

All submissions, including submissions by project members, require review. We use GitHub pull requests for this purpose. Consult GitHub Help for more information on using pull requests.

Quick start with Netlify

Here’s a quick guide to updating the docs. It assumes you’re familiar with the GitHub workflow and you’re happy to use the automated preview of your doc updates:

  1. Fork the Goldydocs repo on GitHub.
  2. Make your changes and send a pull request (PR).
  3. If you’re not yet ready for a review, add “WIP” to the PR name to indicate it’s a work in progress. (Don’t add the Hugo property “draft = true” to the page front matter, because that prevents the auto-deployment of the content preview described in the next point.)
  4. Wait for the automated PR workflow to do some checks. When it’s ready, you should see a comment like this: deploy/netlify — Deploy preview ready!
  5. Click Details to the right of “Deploy preview ready” to see a preview of your updates.
  6. Continue updating your doc and pushing your changes until you’re happy with the content.
  7. When you’re ready for a review, add a comment to the PR, and remove any “WIP” markers.

Updating a single page

If you’ve just spotted something you’d like to change while using the docs, Docsy has a shortcut for you:

  1. Click Edit this page in the top right hand corner of the page.
  2. If you don’t already have an up to date fork of the project repo, you are prompted to get one - click Fork this repository and propose changes or Update your Fork to get an up to date version of the project to edit. The appropriate page in your fork is displayed in edit mode.
  3. Follow the rest of the Quick start with Netlify process above to make, preview, and propose your changes.

Previewing your changes locally

If you want to run your own local Hugo server to preview your changes as you work:

  1. Follow the instructions in Getting started to install Hugo and any other tools you need. You’ll need at least Hugo version 0.146.0 (we recommend using the most recent available version), and it must be the extended version, which supports SCSS.

  2. Fork the Goldydocs repo repo into your own project, then create a local copy using git clone.

    git clone --branch v0.12.0 --depth 1 https://github.com/google/docsy-example.git

  3. Run hugo server in the site root directory. By default your site will be available at http://localhost:1313/. Now that you’re serving your site locally, Hugo will watch for changes to the content and automatically refresh your site.

  4. Continue with the usual GitHub workflow to edit files, commit them, push the changes up to your fork, and create a pull request.

Creating an issue

If you’ve found a problem in the docs, but you’re not sure how to fix it yourself, please create an issue in the Goldydocs repo. You can also create an issue about a specific page by clicking the Create Issue button in the top right hand corner of the page.
